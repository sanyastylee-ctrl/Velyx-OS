# Ключевые security-риски Velyx OS

## 1. Избыточные привилегии GUI-компонентов

Риск:

- shell, settings или store могут стать скрытым обходом модели разрешений

Контрмера:

- UI только инициирует действия
- привилегированные операции уходят в отдельные сервисы
- все sensitive operations централизуются и логируются

## 2. Размывание границы между системой и пользовательскими данными

Риск:

- невозможность безопасного rollback
- рост поверхности атаки

Контрмера:

- разделение system state, app state и user data
- Btrfs snapshots только как часть управляемого recovery design

## 3. Небезопасная модель sideload и Windows-совместимости

Риск:

- запуск неподписанного или вредоносного кода с чрезмерным доступом

Контрмера:

- compatibility manager как обязательный посредник
- trust levels
- отдельные sandbox-профили
- явные предупреждения и ограничение прав

## 4. Сетевые сервисы по умолчанию

Риск:

- лишние открытые порты и фоновая поверхность атаки

Контрмера:

- disabled by default
- включение только по необходимости
- инвентаризация listening services в CI/тестовых профилях позже

## 5. Слабая auditability

Риск:

- невозможно расследовать инциденты или нарушения разрешений

Контрмера:

- централизованный audit/event pipeline
- отдельный журнал security-важных действий

## 6. Неудачные обновления

Риск:

- потеря загрузки, частично обновленная система, нарушение доверия к платформе

Контрмера:

- подписанные обновления
- staged apply
- rollback как обязательный сценарий, а не аварийная импровизация
