# Архитектура Velyx OS

## 1. Архитектурный принцип по умолчанию

Безопасность является обязательным критерием приемки архитектурных решений.

Это означает:

- если решение удобно, но размывает границы привилегий, оно отклоняется
- если решение ускоряет разработку, но ухудшает rollback, audit или sandboxing, оно отклоняется либо помечается как временное с компенсационными мерами
- если совместимость с Windows требует опасного доступа к системе, совместимость ограничивается, а не ядро безопасности

## 2. Выбранная модель платформы

Velyx OS строится на современном Linux-стеке:

- Linux LTS kernel
- Wayland
- PipeWire
- systemd
- Btrfs
- sandbox-first доставка приложений
- атомарные или полуатомарные обновления

## 3. Основные архитектурные решения

### Monorepo на старте

На старте выбран monorepo, потому что:

- shell, settings, files и системные сервисы будут быстро эволюционировать совместно
- дизайн-система и security UX должны версионироваться синхронно
- ранняя стадия продукта выигрывает от единого CI и общей схемы интеграции

### Qt 6 + QML для shell и нативных приложений

Причины:

- зрелый desktop UI стек
- хорошая поддержка Wayland
- сильная модель композиции интерфейса
- удобство построения единой дизайн-системы

### Rust для security-critical и системных сервисов

Причины:

- более безопасная работа с памятью
- хороший баланс скорости и надежности
- удобство построения изолированных сервисов с четкими контрактами

### D-Bus как базовая IPC-шина

Причины:

- нативный desktop/Linux контекст
- понятная модель для разграничения ответственности между сервисами
- удобная интеграция с сессионными и системными службами

## 4. Базовые контуры системы

### Контур A. Управляемое системное ядро

Сюда входят:

- базовый образ системы
- update pipeline
- recovery pipeline
- загрузочная логика
- проверка целостности

Требование:

- immutable или semi-immutable core

### Контур B. Пользовательская среда

Сюда входят:

- shell
- settings
- files
- нативные GUI-приложения

Требование:

- отсутствие постоянных административных прав

### Контур C. Песочница приложений

Сюда входят:

- sandbox policy
- permissions broker
- privacy ledger
- portal-подобные механизмы доступа к ресурсам

Требование:

- deny by default для чувствительных ресурсов

### Контур D. Совместимость Windows

Сюда входят:

- managed runtime
- compatibility profiles
- isolated prefixes
- install mediation

Требование:

- совместимость не должна иметь привилегии, которых нет у нативного sandboxed-приложения без отдельного явного решения

## 5. Security-critical компоненты

- `update-engine`
- `recovery-service`
- `permissions-service`
- `privacy-service`
- `compatibility-manager`
- `app-sandbox-service`
- `secret-store` в будущих фазах
- `audit/logging pipeline`

## 6. Приоритеты безопасности

Приоритет реализации в архитектуре:

1. Целостность системы и безопасные обновления
2. Разделение привилегий и sandbox-модель
3. Recovery и rollback как часть модели доверия
4. Privacy и прозрачность разрешений
5. Безопасная совместимость с Windows
